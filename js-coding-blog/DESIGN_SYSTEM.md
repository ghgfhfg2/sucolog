# JS Coding Blog 디자인 시스템

## 1. 문서 목적
이 문서는 정적 기반 JavaScript 코딩 연습 블로그의 UI 기준을 정의한다.

핵심 목표는 다음 세 가지다.

1. 문제를 읽기 쉽다
2. 코드를 쓰기 편하다
3. 실행 결과를 빠르게 이해할 수 있다

즉, 장식보다 **집중감, 명확성, 구현 용이성**을 우선한다.

---

## 2. 제품 톤 앤 무드
이 프로젝트의 시각적 방향은 아래와 같다.

- 조용한 개발자 감성
- 다크 베이스
- 실용적이고 과장 없는 UI
- 문제 풀이에 집중되는 인터페이스

키워드:

- Focused
- Technical
- Calm
- Minimal
- Readable

---

## 3. 디자인 원칙

### 3.1 읽는 흐름이 먼저
문제 설명, 제한사항, 예시, 코드 입력, 결과가 자연스럽게 이어져야 한다.

### 3.2 코드가 주인공
페이지에서 가장 중요한 인터랙션은 코드 작성이다. 에디터 영역은 충분히 크고 안정적이어야 한다.

### 3.3 상태는 즉시 이해 가능해야 함
실행 결과는 복잡한 장식 없이도 PASS/FAIL, 오류 여부가 바로 보여야 한다.

### 3.4 확장 가능한 최소 구조
처음에는 textarea 기반으로 단순하게 가되, 이후 Monaco Editor, 힌트/해설 UI를 붙이기 쉬운 구조를 유지한다.

---

## 4. 정보 구조
문제 상세 페이지 기본 순서:

1. 문제 제목
2. 메타 정보 (난이도, 카테고리, 함수명)
3. 문제 설명
4. 제한사항
5. 예시
6. 코드 입력 영역
7. 실행 버튼 / 초기화 버튼
8. 결과 영역

---

## 5. 레이아웃 시스템

### 5.1 전체 레이아웃
- 최대 너비: 960px
- 중앙 정렬
- 상단 고정 헤더
- 단일 컬럼 우선

### 5.2 섹션 구조
주요 영역은 카드/패널 단위로 분리한다.

- Problem Content Card
- Code Editor Card
- Result Card

### 5.3 반응형 원칙
- **데스크톱:** 단일 컬럼, 넓은 에디터, 넉넉한 여백
- **모바일:** 패딩 축소, 버튼 줄바꿈 허용, 에디터 높이 유지

### 6.2 Typography
```css
--font-body: -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
--font-mono: ui-monospace, SFMono-Regular, Menlo, monospace;
```

권장 크기:
- H1: 32px
- H2: 22px
- Body: 16px
- Small: 14px
- Caption: 13px

### 6.3 Spacing

```css

```

## 7. 핵심 컴포넌트

### 7.1 Header
역할:
- 브랜드 표시
- 홈 / 문제 목록 이동

원칙:
- 작고 단정하게
- 본문보다 튀지 않게
- sticky 허용

### 7.2 Problem Meta Pill
용도:
- 난이도
- 카테고리
- 함수명

원칙:
- 작은 태그처럼 보여야 함
- 지나치게 강한 강조는 피함

### 7.3 Problem Content Card
포함:
- 설명
- 제한사항
- 예시
- 인라인 코드

원칙:
- 텍스트 가독성 우선
- 코드 조각은 가볍게 강조

### 7.4 Editor
초기 버전:
- textarea 기반

원칙:
- 충분한 높이
- monospace 폰트
- 어두운 배경
- 명확한 경계

### 7.5 Buttons
종류:
- Primary: 실행
- Secondary: 초기화

원칙:
- 실행 버튼이 항상 더 눈에 띄어야 함
- hover/focus 상태 제공

### 7.6 Result Panel
표현 요소:
- 총 통과 수
- 테스트별 상태
- 입력값
- 기대값
- 실제값
- 오류 메시지

원칙:
- 초기에는 텍스트 기반으로 단순하게 유지
- 이후 카드형 결과 리스트로 확장 가능

---

## 8. 상태 시스템
명확히 구분할 상태:

- 아직 실행하지 않음
- 전체 테스트 통과
- 일부 테스트 실패
- 실행 오류

색상과 텍스트를 함께 사용한다. 색만으로 상태를 표현하지 않는다.

---

## 9. 접근성 원칙
- 텍스트 대비 충분히 확보
- 버튼과 입력창 focus 상태 제공
- 키보드만으로 실행 가능
- 결과는 텍스트로도 충분히 설명
- 너무 작은 클릭 영역 금지

---

## 10. 보안 관점의 UI 원칙
- 사용자 입력은 HTML처럼 렌더링하지 않는다
- 출력은 텍스트 중심으로 보여준다
- 설명 영역 / 입력 영역 / 결과 영역을 시각적으로 분리한다
- 코드 실행 맥락과 콘텐츠 표시 맥락이 혼동되지 않게 한다

---

## 11. MVP 범위
포함:
- 다크 톤 기반 UI
- 공통 헤더
- 문제 목록 UI
- 문제 상세 UI
- textarea 에디터
- 실행 버튼 / 초기화 버튼

제외:
- 고급 애니메이션
- 사용자 테마 설정
- 랭킹/점수 UI
- 다중 언어 UI
- 복잡한 대시보드

## 12. 우선순위

- 버튼 스타일 정리

### 2순위

### 3순위
- Monaco Editor 도입

- 힌트/해설 컴포넌트 추가

## 13. 결론
이 프로젝트의 디자인 시스템 방향은 다음과 같다.

**다크 기반, 개발자 친화적, 실용적이고 확장 가능한 최소 UI 시스템**

핵심은 예쁨보다 문제 이해, 코드 작성, 결과 해석의 흐름을 최대한 부드럽게 만드는 것이다.
